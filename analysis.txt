## CRITICAL BUGS IDENTIFIED

### BUG #1: DEEP COPY ISSUE IN GLOBAL SETTINGS (SCENARIO A)
**File**: global_settings.py, lines 57, 429

In __init__:
`python
self._contour_copy: Dict[str, Dict] = deepcopy(state.contour_layer_configs)
`

In _write_to_state (line 429):
`python
s.contour_layer_configs = deepcopy(self._contour_copy)
`

**THE PROBLEM**: 
- _contour_copy is a deepcopy of state.contour_layer_configs
- When _save_current_contour() runs (lines 352-370), it calls:
  `python
  self._contour_copy[key] = {...}
  `
  This CREATES A NEW DICT - it doesn't update the old one via reference
- Each time a user changes ANY contour control, a new dict is created in _contour_copy
- BUT _load_from_state only runs ONCE in __init__ (line 62)
- So if the user opens Global Settings, then changes opacity in Display Settings panel, 
  THEN opens Global Settings again, the dialog won't see the changes from the panel!

**Timeline**:
1. User opens Global Settings → _load_from_state() → _contour_copy = deepcopy(state)
2. User changes opacity to 0.5 in Display Settings PANEL
3. state.contour_layer_configs['surface']['opacity'] = 0.5
4. state.display_settings_changed.emit()
5. User opens Global Settings again
6. _load_from_state() is NOT called again! (it only runs in __init__)
7. _contour_copy still has old value (0.7)
8. Dialog shows stale opacity value

**VERIFICATION**: Line 292-310 in global_settings.py:
- _load_from_state() is called only in __init__, never after dialog re-opens
- No refresh mechanism exists for when dialog is shown again

### BUG #3: NO REFRESH OF LAYER PANEL WHEN GLOBAL SETTINGS CLOSES
**File**: global_settings.py (no connection), layer_panel.py line 193

Layer panel connects to display_settings_changed (line 193):
`python
self.state.display_settings_changed.connect(self._refresh_disp_widgets)
`

BUT:
- global_settings.py DOES emit display_settings_changed (line 436)
- HOWEVER, _refresh_disp_widgets() (layer_panel.py lines 353-355) calls _load_disp_layer()
- _load_disp_layer() reads from state and updates UI widgets
- BUT it doesn't BLOCK SIGNALS while updating
- So when layer panel's widgets change, it triggers _on_disp_changed() AGAIN
- And _on_disp_changed() emits display_settings_changed AGAIN
- This could cause cascading updates!

**Check layer_panel.py _refresh_disp_widgets**:
`python

    """Re-load display widgets when state changes externally."""
    self._load_disp_layer()

def _load_disp_layer(self, _idx=0):

    self._disp_loading = True  # Guard set
    key = self.disp_layer.currentData()
    cfg = self.state.get_contour_config(key)
    self.opacity_slider.setValue(int(cfg['opacity'] * 100))  # Triggers valueChanged!
    ...
    self._disp_loading = False
`

- Line 331 sets _disp_loading = True BEFORE changing widgets
- Line 334 changes slider (will trigger valueChanged signal)
- Line 342 sets _disp_loading = False
- **BUT**: The valueChanged signal will call _on_disp_changed()
- _on_disp_changed() checks if self._disp_loading: return (line 342)
- So the guard SHOULD work... but the guard is checked AFTER the signal is emitted
- Actually, looking at PyQt: when you call setValue(), it emits the signal immediately
- The guard IS in place, so this shouldn't cascade

**However**, there's still an issue:
- If global_settings changes multiple settings, display_settings_changed is emitted ONCE
- But the layer panel reads ALL settings when refreshed
- If the user had manually edited a setting in the panel that global_settings didn't change...
- It will be OVERWRITTEN by the refresh!

Example:
1. User sets opacity=0.5 in panel, levels=20
2. User opens Global Settings, changes ONLY colorscale
3. Global Settings calls _write_to_state() which does:

4. But _contour_copy was loaded from state in __init__, which had opacity=0.7, levels=15!
5. So the panel's changes (opacity=0.5) are LOST!

This is the BUG: **Global Settings doesn't merge its changes with existing state, it REPLACES the entire dict**

**File**:

**THERE'S A 300ms DELAY!**

Timeline:
1. User changes opacity in layer panel
2. _on_disp_changed() runs, sets value in state, emits display_settings_changed
3. map_widget._schedule_refresh() starts a 300ms debounce timer
4. User immediately opens Global Settings before timer fires
5. Global Settings dialog reads from state (which has the new opacity) ✓
6. User changes opacity in Global Settings
7. _save_current_contour() updates _contour_copy ✓
8. User clicks OK

12. BUT in parallel, the old timer might STILL be running!
13. When old timer fires, refresh() reads state and renders with OLD opacity
14. Then new timer fires, refresh() reads state and renders with NEW opacity
15. So you see TWO renders, but the first one is with stale data!

**WORSE**: If the user has NOT touched the Global Settings dialog:
1. Panel changes opacity=0.5
2. display_settings_changed emitted
3. map_widget starts 300ms debounce

5. display_settings_changed emitted AGAIN
